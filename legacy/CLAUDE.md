# CLAUDE.md — MdLaw 프로젝트 AI 작업 지침

> 이 파일은 Claude가 이 프로젝트를 작업할 때 항상 따라야 하는 규칙입니다.
> 대화 시작 시 HANDOFF.md + PROGRESS.md와 함께 공유하면 바로 맥락을 파악합니다.

---

## 1. 세션 시작 시 반드시 할 것

1. `HANDOFF.md` 읽기 → 현재 Phase, 핵심 파일, 환경 변수 파악
2. `PROGRESS.md` 읽기 → 마지막 작업 내역 및 알려진 한계 파악
3. 요청된 작업 범위 확인 후 작업 시작

---

## 2. 코드 수정 원칙 — "핀포인트 수정"

### ✅ 허용
- 명시적으로 요청된 파일/라인만 수정
- 요청된 버그 수정, 기능 추가 범위 내 최소 변경

### ❌ 금지
- 요청하지 않은 파일 수정 (예: 버그 수정 요청 시 관련 없는 컴포넌트 변경 금지)
- 리팩토링, 변수명 변경, 코드 스타일 통일 (별도 요청 없으면 하지 않음)
- 작동 중인 기능 임의 변경
- `USE_DUMMY` 플래그 변경 금지 (현재: `false` 유지)
- 환경 변수 값 변경 금지

### 애매한 경우
- 수정 범위가 불명확하면 먼저 어떤 파일을 수정할지 설명하고 확인받기

---

## 3. HANDOFF.md 업데이트 규칙

### 업데이트 시점
- 새로운 Phase가 완료될 때마다
- 핵심 파일 역할이 바뀔 때 (예: search/route.js가 Supabase로 전환)
- 새로운 환경 변수가 추가될 때
- 알려진 한계가 해소되거나 새로 생길 때

### 수정 위치
```
HANDOFF.md 구조 (이 순서 유지):
─────────────────────────────────
[헤더 + 마지막 업데이트 날짜]
[프로젝트 빠른 시작]
[Phase 완료 섹션 — 최신 Phase가 위, 오래된 Phase가 아래]  ← 순서 중요
[핵심 파일 위치]
[환경 변수]
[검색 구조 변천사]
[법제처 API 정보]
[과거 버그 기록]
[알려진 한계]
─────────────────────────────────
```

### Phase 번호 규칙
- 새 Phase는 이전 Phase 번호 + 1
- Phase 섹션은 항상 최신이 위, 오래된 것이 아래 (역순 정렬)
- 형식: `## ✅ Phase N 완료 (YYYY-MM-DD)`

### 마지막 업데이트 날짜
- 세션에서 파일을 수정한 경우 반드시 갱신: `**마지막 업데이트: YYYY-MM-DD**`

---

## 4. PROGRESS.md 업데이트 규칙

### 업데이트 시점
- 작업을 완료한 모든 세션 후

### 형식
```markdown
## ✅ 완료 (YYYY-MM-DD) — [작업 제목]

### 문제 (있는 경우)
- 증상 설명

### 원인
- 기술적 원인

### 수정 (핀포인트)
| 파일 | 변경 내용 |
|---|---|
| `경로/파일.js` | 변경 요약 |
```

### 순서
- 새 항목은 항상 파일 최상단에 추가 (최신이 위)
- 기존 항목은 절대 삭제하지 않음 (버그 재발 방지 목적)

---

## 5. 배포 관련 주의사항

- 배포: Vercel 자동 배포 (main 브랜치 push 시 자동)
- 배포 전 로컬 확인 불필요 (Next.js edge runtime 환경 차이 있음)
- `export const runtime = "edge"` — 모든 API route에 유지
- `export const preferredRegion = "icn1"` — 법제처 API는 한국 IP 필요, 반드시 유지

---

## 6. 절대 건드리지 않는 것

| 항목 | 이유 |
|------|------|
| `.env.local` 파일 | 민감 정보 |
| `USE_DUMMY = false` | Phase 2 이후 항상 false |
| `preferredRegion = "icn1"` | 법제처 API 한국 IP 필수 |
| `Referer` 헤더 | 법제처 API 차단 방지 |
| `collect-laws.mjs` Supabase 적재 로직 | 데이터 무결성 |
